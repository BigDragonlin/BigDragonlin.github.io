# 多个Servlet调用的规则

多个servlet的调用规则：

  • 重定向解决方案
  • 请求转发解决方案

# 重定向解决方案

工作原理：通过浏览器自动完成重定向

实现命令：response.sendRedirect

重定向的特征：

  • 请求地址：既可以是外部网站地址，也可以是内部网站的地址
  • 请求次数：浏览器至少完成两次请求，一次是用户手动请求，一次是浏览器接受302代码后自动重定向
  • 请求方式:只要是通过地址栏跳转的都是get方式
  • 缺点：重定向比较浪费时间；

# 请求转发解决方案

  • 原理：用户第一次通过手动的方式请求服务器访问OneServlet，OneServlet完成工作后，当前的请求对象代替浏览器来请求tomcat服务器访问TwoServlet，tomCat接到请求后自动调用TwoServlet完成剩余任务。
  • 实现命令：request.getRequestDispatcher.forword(request,response)
  • 优点:无论服务器中间经过几次使用serverlet中转，用户只需要在浏览器上发一回请求。节省服务端和浏览器之间不断重定向需要的时间
  • 特征：浏览器只需要发送一回请求；request.getRequestDispatcher.forword不能和重定向方式一样访问外部资源文件，只能访问内部的文件

# HttpSession接口

  1. 介绍

    • 实现的servlet的一个接口；TomCat实现了Servlet.api接口
    • 数据共享；如果两个Servlet来自于同一个网站，且两个servlet被同一个用户/浏览器访问，那么我们可以使用HttpServlet来实现数据的共享；
    • 会话作用域对象；中国程序员习惯把HttpServlet对象称之为会话作用域对象，把session称之为会话

  2. httpsession 与 cookie的区别【面试】

    • 存储位置不同；一个在云上，一个在地下，HttpSession储存的信息都是在服务器的内存中，而Cookie的储存信息都在用用户本地的内存/硬盘中

    • 存储数据类型不同。HttpSeession可以存储object类型的信息，而Cookie只能存储String类型的信息

    • 数据数量不同。cookie只能一次放一个键值对。httpsession使用map数据，可以存任意数量的数据

    • 参照物

      cookie相当于会员卡

      httpSession相当于【私人保险柜】。http产生是因为cookie存放的数据类型过于单一，数量太少

  3. 命令实现

    • 开启session

      HttpSession session = request.getHttpSession

    • 设置session的属性

      session.getAttribute

      session.setAttribute

  4. Http服务器如何将用户与HttpSession关联起来

    使用Cookie可以把使得http服务器得到每个用户和浏览器的手牌，从而识别用户身份

  5. getSession（） 与 getSession（false）

    • getSession():

      如果当前浏览器或者用户有手牌那么就会返回手牌的id，如果当前浏览器或者用户没有手牌，那么就给当前浏览器或者用户创建一个手牌

    • getSession(false):

      如果当前浏览器或者用户有手牌，那么就会返回当前用户或者浏览器的id，如果当前用户或者浏览器没有被创建TomCat服务器就会返回null

    这两个方式使用什么情景

    getSession（）在已经确认用户身份的时候使用；getSession（false）在没有确认用户身份的时候使用

  6. HttpSession的销毁时机

    用户与HttpSession关联使用的Cookie存放在浏览器的缓存中，当用户关闭浏览器，缓存清空，此时HttpSession和用户的关系切断。

    由于TomCat并不知道我们什么时候关闭浏览器，所以TomCat中默认最后一次使用HttpSession的30分钟后销毁HttpSession。

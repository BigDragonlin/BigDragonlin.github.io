# GitHub merge

  • create a merge commit

    • squash 在这里是压缩的意思
  • rebase
    • rebase 中文翻译过来就是变基
    • git rebase main
    • feature 分支消失，合并到 main 分支
操作 rebase merge squash merge
特点 只有线性提交记录 会出现所有提交记录，包括 merge 记录 只出现一条 merge 记录
优点 减少一次 merge 记录 保证分支可溯源 历史记录更加清爽干净
缺点 必须使用强推 多一次 merge 记录 历史记录被合并到了一起
应用场景 在私有分支上可以用 rebase 多人协作分支必须 merge，向主干分支合并必须 merge 多人协作分支，需要保持记录清爽

user: 这几个开源协议有什么区别 [Timestamp: 2025/12/13 13:42:53]

ai: 这几个主要的开源协议在核心要求、限制和“传染性”上有显著区别。以下是它们的核心区别对比：

协议 核心要求 限制与“传染性” 典型使用场景
GPL (GNU General Public License) 1. 开源代码
2. 修改后的衍生作品也必须开源		 强传染性：任何包含或修改了GPL代码的软件，其整个衍生作品都必须以GPL开源。 强调软件自由，希望所有衍生作品都保持开源的项目（如Linux内核、GIMP）。
LGPL (GNU Lesser General Public License) 1. 开源代码
2. 对库本身的修改必须开源		 弱传染性：仅当直接修改了LGPL库的代码时，修改部分需开源。如果只是动态链接使用该库，则不要求主程序开源。 希望被广泛采用（包括闭源软件）的库（如GTK+、FFmpeg库）。
MIT License 1. 保留原协议和版权声明 无传染性：允许闭源、修改、分发、商业使用，且衍生作品无需开源。 非常宽松，希望最大限度被广泛使用的项目（如jQuery、Node.js、React）。
BSD License 1. 保留原协议和版权声明
2. 不得使用原作者名称为衍生作品背书		 无传染性：与MIT类似，允许闭源和商业使用，衍生作品无需开源。BSD条款更具体一些。 宽松开源，常见于学术和商业软件（如FreeBSD、NetBSD）。
Apache License 2.0 1. 保留原协议和版权声明
2. 修改文件需注明
3. 提供专利授权（明确专利许可）		 无传染性：允许闭源和商业使用，衍生作品无需开源。提供了明确的专利保护条款。 企业级项目，重视专利保护和明确法律条款（如Apache HTTP Server、Android、Kubernetes）。

# 快速选择指南

  • 想最大程度推广，不介意别人闭源：选 MIT（最简）或 Apache 2.0（有专利条款）。
  • 开发库，希望被闭源软件使用：选 LGPL 或 MIT/Apache。
  • 坚持所有衍生作品都必须开源：选 GPL。
  • 企业项目，需要明确的专利保护：选 Apache 2.0。

简单来说，从限制严格到最为宽松的排序大致是：GPL > LGPL > Apache ≈ BSD > MIT。其中“传染性”是区分GPL家族与其他协议的关键。 [Timestamp: 2025/12/13 13:43:15]

# GitHub 做开源项目的基本流程

在 GitHub 上发起和维护一个开源项目，通常遵循以下标准流程，这有助于项目健康成长并吸引贡献者。

# 1. 项目初始化

  • 创建仓库
    • 在 GitHub 上点击 “New repository”。
    • 填写仓库名、描述，选择公开（Public）。
    • 强烈建议：初始化 README.md、.gitignore（选择项目语言）和许可证（License）。
  • 本地关联与首次提交
    git clone <your-repo-url>
cd <repo-name>
# 进行初始开发...
git add .
git commit -m "Initial commit"
git push -u origin main

# 2. 项目规范化（关键步骤）

一个规范的项目能极大降低贡献者的参与门槛。

  • 编写清晰的 README.md：项目的“门面”，应包含：
    • 项目名称与简介
    • 快速安装和使用指南
    • 功能特性列表
    • 贡献指南的链接
  • 选择开源协议：在根目录添加 LICENSE 文件。明确告知他人如何使用你的代码（如 MIT, GPL, Apache 2.0）。
  • 设置贡献指南：创建 CONTRIBUTING.md 文件，说明：
    • 如何报告 Bug（Issue 模板）
    • 如何提议新功能
    • 代码提交流程（如 Fork & Pull Request）
  • 制定行为准则：大型或社区项目可添加 CODE_OF_CONDUCT.md（如贡献者公约），营造友好环境。
  • 完善文档：考虑使用 docs 文件夹或 Wiki 来存放详细文档。

# 3. 协作流程（GitHub Flow）

这是 GitHub 上最核心的协作模式。

  1. 基于 Issue 开发：任何新功能、Bug 修复都应先创建 Issue 进行讨论。
  2. 从主分支创建特性分支：
    git checkout main
git pull origin main
git checkout -b feature/awesome-new-feature
    1
    2
    3
  3. 在分支上进行开发并提交。
  4. 推送分支并创建 Pull Request：
    • 将分支推送到远程仓库：git push -u origin feature/...
    • 在 GitHub 仓库页面会提示创建 Pull Request。
  5. PR 审查与讨论：
    • 在 PR 中描述更改内容，链接相关 Issue（使用 # 加 Issue 号）。
    • 其他贡献者或维护者进行代码审查，提出修改意见。
  6. 合并与部署：
    • 通过所有审查后，由维护者将 PR 合并（Merge）到主分支。
    • 可以使用“合并提交”、“变基合并”或“压缩合并”。
    • 理想情况下，主分支应始终保持可部署状态。

# 4. 维护与管理

  • 管理 Issues：定期处理，使用标签（Labels）进行分类（如 bug， enhancement， help wanted）。
  • 处理 Pull Requests：积极回复，给予新人清晰的反馈。
  • 发布版本：使用 Git 标签（Tag）标记发布版本。
    git tag -a v1.0.0 -m "Release version 1.0.0"
git push origin --tags
    1
    2
  • 利用 GitHub 功能：
    • Projects：看板管理项目进度。
    • Actions：设置 CI/CD 流水线，自动化测试和部署。
    • Wiki：维护项目详细文档。
    • Discussions：用于开放式问答和社区讨论。

# 5. 推广与社区建设

  • 在项目简介中使用相关主题（Topics）。
  • 在技术社区、社交媒体上分享项目。
  • 积极回应 Issue 和 PR，鼓励首次贡献者。
  • 对于重要贡献者，可以邀请其成为项目的协作者（Collaborator）。

# 核心原则

  • 开放透明：所有讨论（Issue, PR）公开进行。
  • 文档优先：良好的文档是项目成功的基石。
  • 流程化：固定的流程（如 Fork-PR）能提高协作效率。
  • 友善包容：维护健康的社区氛围。

遵循这个基本流程，可以帮助你的开源项目从个人项目顺利过渡到受人欢迎的社区项目。 [Timestamp: 2025/12/13 13:56:04]
